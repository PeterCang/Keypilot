# KeyPilot — AI 开发工具 API Key 管理器调研报告

**项目名称**：KeyPilot  
**工具命令**：`KeyPilot`  
**调研日期**：2026-05-15  
**调研背景**：Claude Code CLI 和 OpenAI Codex CLI/App 在使用 API Key 时，切换 Key 的流程繁琐，需要手动修改配置文件或环境变量并重启服务。本报告调研开发 KeyPilot 的可行性与技术方案。KeyPilot 同时承担 AI 开发工具的安装与管理职责。

**数据存储原则**：所有数据仅保存在本地，以 JSON 明文格式存储，不依赖系统密钥库，不上传任何数据到远端。

---

## 一、现状分析

### 1.1 痛点描述

当前使用 Claude Code 和 Codex 等 AI 开发工具时，API Key 管理存在以下问题：

1. **切换繁琐**：额度耗尽后需手动修改配置文件或环境变量
2. **需要重启**：修改配置后必须重启工具进程才能生效
3. **多 Key 管理困难**：Key 数量多时，记忆和管理成本高
4. **跨工具不统一**：Claude Code 和 Codex 的配置路径、格式各不相同

### 1.2 KeyPilot 定义

开发一款跨平台 UI 工具 **KeyPilot**，具备以下能力：

- 支持 Windows 和 macOS
- 针对 Claude Code、Codex、Gemini CLI 等工具的 Key 进行添加、编辑、删除管理
- 一键切换当前使用的 Key
- 支持安装 / 更新 AI 开发工具（Codex App、Codex CLI、Claude Code、Gemini CLI 等）
- 所有数据本地存储，JSON 明文格式，用户可直接查看和备份

## 二、配置机制调研

### 2.1 Claude Code CLI

Claude Code 通过**环境变量**配置 Key 和 Base URL，不使用独立的凭证配置文件。

| 配置项 | 详情 |
|---|---|
| API Key 环境变量 | `ANTHROPIC_AUTH_TOKEN` |
| Base URL 环境变量 | `ANTHROPIC_BASE_URL` |
| 全局配置目录 | `~/.claude/` |
| 行为设置文件 | `~/.claude/settings.json` |
| 项目级配置 | `.claude/settings.json`（项目根目录，优先级高于全局） |
| 热重载支持 | 不支持，需重启进程 |

#### 安装方式

| 系统 | 命令 |
|---|---|
| Windows | `irm https://claude.ai/install.ps1 \| iex`（PowerShell） |
| macOS | `curl -fsSL https://claude.ai/install.sh \| bash` |

#### macOS 环境变量配置

```bash
# zsh（默认）
echo 'export ANTHROPIC_BASE_URL="https://your-endpoint.com"' >> ~/.zshrc
echo 'export ANTHROPIC_AUTH_TOKEN="your-api-key"' >> ~/.zshrc
source ~/.zshrc

# bash
echo 'export ANTHROPIC_BASE_URL="https://your-endpoint.com"' >> ~/.bash_profile
echo 'export ANTHROPIC_AUTH_TOKEN="your-api-key"' >> ~/.bash_profile
source ~/.bash_profile
```

#### Windows 环境变量配置（PowerShell）

```powershell
[System.Environment]::SetEnvironmentVariable('ANTHROPIC_BASE_URL', 'https://your-endpoint.com', 'User')
[System.Environment]::SetEnvironmentVariable('ANTHROPIC_AUTH_TOKEN', 'your-api-key', 'User')
```

> 注意：Windows 设置用户级环境变量后，需重新启动 PowerShell / CMD 或注销重新登录才能生效。

配置优先级：**环境变量 > 项目配置 > 全局配置**

### 2.2 OpenAI Codex CLI / App（Mac）

Codex 采用**双文件分离设计**，配置与凭证解耦：

| 配置项 | 详情 |
|---|---|
| 全局配置目录 | `~/.codex/` |
| 模型与 Provider 配置 | `~/.codex/config.toml` |
| API Key 凭证文件 | `~/.codex/auth.json` |
| 热重载支持 | 不支持，需重启 CLI 或 App |

`~/.codex/config.toml` 典型结构：

```toml
model_provider = "gac"
model = "gpt-5.5"
model_reasoning_effort = "medium"
disable_response_storage = true
preferred_auth_method = "apikey"

[model_providers.gac]
name = "gac"
base_url = "https://gaccode.com/codex/v1"
wire_api = "responses"
```

`~/.codex/auth.json` 结构：

```json
{
  "OPENAI_API_KEY": "your-api-key"
}
```

**设计要点**：
- `model_provider` 指定当前使用的 provider 名称，对应 `[model_providers.{name}]` 块
- `base_url` 支持自定义 API 端点，可对接第三方兼容服务
- Key 单独存放在 `auth.json`，切换 Key 只需修改该文件，无需动 `config.toml`
- 切换 provider（base_url）只需修改 `config.toml`，两个操作完全独立

#### Codex 安装方式

| 形式 | 安装方式 |
|---|---|
| Codex App（Windows） | 下载地址：`https://get.microsoft.com/installer/download/9PLM9XGG6VKS?cid=website_cta_psi` |
| Codex CLI | `npm i -g @openai/codex` |

### 2.3 Google Gemini CLI

Gemini CLI 同样通过**环境变量**配置，无独立配置文件。

| 配置项 | 详情 |
|---|---|
| API Key 环境变量 | `GEMINI_API_KEY` |
| Base URL 环境变量 | `GOOGLE_GEMINI_BASE_URL` |
| 模型环境变量 | `GEMINI_MODEL` |
| 热重载支持 | 不支持，需重启进程 |

#### 安装方式

```bash
npm install -g @google/gemini-cli
# 要求 Node.js 20.0.0+
```

#### macOS 环境变量配置

```bash
# zsh（默认）
echo 'export GOOGLE_GEMINI_BASE_URL="https://your-endpoint.com"' >> ~/.zshrc
echo 'export GEMINI_API_KEY="your-api-key"' >> ~/.zshrc
echo 'export GEMINI_MODEL="gemini-2.5-pro"' >> ~/.zshrc
source ~/.zshrc

# bash
echo 'export GOOGLE_GEMINI_BASE_URL="https://your-endpoint.com"' >> ~/.bash_profile
echo 'export GEMINI_API_KEY="your-api-key"' >> ~/.bash_profile
echo 'export GEMINI_MODEL="gemini-2.5-pro"' >> ~/.bash_profile
source ~/.bash_profile
```

#### Windows 环境变量配置（PowerShell）

```powershell
[System.Environment]::SetEnvironmentVariable("GOOGLE_GEMINI_BASE_URL", "https://your-endpoint.com", [System.EnvironmentVariableTarget]::User)
[System.Environment]::SetEnvironmentVariable("GEMINI_API_KEY", "your-api-key", [System.EnvironmentVariableTarget]::User)
[System.Environment]::SetEnvironmentVariable("GEMINI_MODEL", "gemini-2.5-pro", [System.EnvironmentVariableTarget]::User)
```

### 2.4 跨平台配置路径对比

| 工具 | 配置方式 | macOS 路径 | Windows 路径 |
|---|---|---|---|
| Claude Code | 环境变量 | `~/.zshrc` / `~/.bash_profile` | 用户级注册表环境变量 |
| Codex CLI/App | 文件 | `~/.codex/config.toml` + `~/.codex/auth.json` | `%USERPROFILE%\.codex\` |
| Gemini CLI | 环境变量 | `~/.zshrc` / `~/.bash_profile` | 用户级注册表环境变量 |

**关键差异**：Claude Code 和 Gemini CLI 依赖环境变量，Codex 依赖配置文件。管理工具需要针对两种机制分别实现写入逻辑。

## 三、技术方案

### 3.1 最终采用方案

**最终方案**：采用 **Tauri + React + TypeScript + Rust 后端 + 系统托盘**。

KeyPilot 定位为跨平台桌面工具，主界面负责 Key 管理、工具状态查看和安装管理；系统托盘负责日常快速切换；Rust 后端负责本地文件读写、环境变量写入、进程检测、工具安装命令执行和日志流式回传。

| 层级 | 技术选型 | 职责 |
|---|---|---|
| 桌面壳 | Tauri | 跨平台打包、窗口管理、系统托盘、原生能力调用 |
| 前端 | React + TypeScript | Key 管理界面、工具状态面板、安装日志、切换确认交互 |
| UI 组件 | shadcn/ui 或 Radix UI + Tailwind CSS | 构建简洁、可维护的桌面工具界面 |
| 后端 | Rust | 文件系统操作、配置写入、环境变量写入、进程检测、子进程执行 |
| 本地存储 | JSON 明文文件 | 保存 Key、工具配置、当前激活项和用户偏好 |
| 打包发布 | Tauri Bundler | Windows MSI/NSIS、macOS DMG/App |

### 3.2 核心架构

```text
KeyPilot
├─ 前端界面（React）
│  ├─ Key 列表管理
│  ├─ 工具状态检测
│  ├─ 一键切换
│  ├─ 安装 / 更新入口
│  └─ 日志输出面板
│
├─ Tauri 命令层
│  ├─ list_keys()
│  ├─ save_key()
│  ├─ switch_key()
│  ├─ detect_tools()
│  ├─ install_tool()
│  └─ open_data_file()
│
└─ Rust 核心模块
   ├─ storage：读写 KeyPilot data.json
   ├─ adapters：Claude / Codex / Gemini 配置适配器
   ├─ env：跨平台环境变量写入
   ├─ process：进程检测与重启引导
   ├─ installer：工具安装与日志流式输出
   └─ backup：配置备份与原子写入
```

### 3.3 模块设计

| 模块 | 说明 |
|---|---|
| `storage` | 管理 `%APPDATA%\KeyPilot\data.json` 和 macOS Application Support 下的数据文件 |
| `tool_adapter` | 抽象不同 AI 工具的配置写入逻辑，统一提供 `detect`、`switch_key`、`backup` 方法 |
| `claude_adapter` | 写入 `ANTHROPIC_AUTH_TOKEN`、`ANTHROPIC_BASE_URL` 等环境变量 |
| `codex_adapter` | 修改 `~/.codex/config.toml` 与 `~/.codex/auth.json` |
| `gemini_adapter` | 写入 Gemini CLI 所需环境变量 |
| `installer` | 检测 Node/npm/目标工具版本，并执行安装或更新命令 |
| `tray` | 托盘菜单展示当前工具与 Key 列表，支持快速切换 |
| `logger` | 将安装、切换、备份、错误信息统一输出到 UI |

### 3.4 一键切换流程

```text
用户选择目标 Key
  ↓
读取 KeyPilot data.json
  ↓
根据 tool 字段选择适配器
  ↓
备份目标工具原配置
  ↓
写入目标配置文件或用户级环境变量
  ↓
更新 data.json 中的 isActive 状态
  ↓
检测目标工具是否正在运行
  ↓
提示用户重启，或执行“切换并重启”
```

### 3.5 Tauri 命令接口设计

```ts
type ToolType = "claude-code" | "codex" | "gemini-cli";

type KeyRecord = {
  id: string;
  name: string;
  tool: ToolType;
  apiKey: string;
  baseUrl?: string;
  model?: string;
  isActive: boolean;
  createdAt: string;
  updatedAt?: string;
  note?: string;
};
```

| 命令 | 参数 | 返回 | 说明 |
|---|---|---|---|
| `list_keys` | 无 | `KeyRecord[]` | 读取本地 Key 列表 |
| `save_key` | `KeyRecord` | `KeyRecord` | 新增或更新 Key |
| `delete_key` | `id` | `boolean` | 删除 Key |
| `switch_key` | `id` | `SwitchResult` | 写入目标工具配置并激活 Key |
| `detect_tools` | 无 | `ToolStatus[]` | 检测 Claude / Codex / Gemini 安装状态 |
| `install_tool` | `tool` | 日志事件流 | 执行安装命令并实时返回日志 |
| `backup_config` | `tool` | `BackupResult` | 手动备份目标工具配置 |

### 3.6 开发优先级

1. **MVP 第一阶段**：本地 JSON 存储、Key CRUD、Codex 配置切换、Claude 环境变量切换。
2. **MVP 第二阶段**：系统托盘快速切换、Gemini CLI 支持、配置备份和失败回滚。

## 四、核心技术难点

### 4.1 环境变量写入机制（Claude Code / Gemini CLI 的核心难点）

**问题**：Claude Code 和 Gemini CLI 通过环境变量传递 Key，而环境变量的写入和生效机制在两个平台上差异显著。

**macOS 写入方案**：

修改 Shell 配置文件（`~/.zshrc` 或 `~/.bash_profile`）：
1. 读取文件内容，检测是否已存在对应 `export` 行
2. 若存在则用正则替换旧值，若不存在则追加新行
3. 修改后提示用户执行 `source ~/.zshrc` 或重启终端

**Windows 写入方案**：

通过 PowerShell 调用 `[System.Environment]::SetEnvironmentVariable()` 写入用户级注册表（`HKCU\Environment`）：
- 新启动的进程自动继承，无需重启系统
- 已运行的进程（包括 Claude Code / Gemini CLI）需重启后才能感知
- 验证方式：`echo $env:ANTHROPIC_AUTH_TOKEN`

**Codex 无此问题**：Key 存储在 `~/.codex/auth.json`，直接文件写入即可，无需处理环境变量。

### 4.2 KeyPilot 本地数据存储设计

**决策**：所有数据本地 JSON 明文存储，不使用系统密钥库（Keychain / Credential Manager）。

**存储文件路径**：

| 系统 | 路径 |
|---|---|
| Windows | `%APPDATA%\KeyPilot\data.json` |
| macOS | `~/Library/Application Support/KeyPilot/data.json` |

**`data.json` 数据结构设计**：

```json
{
  "version": 1,
  "keys": [
    {
      "id": "uuid-1",
      "name": "GAC 主力 Key",
      "tool": "claude-code",
      "apiKey": "sk-ant-xxxxxxxx",
      "baseUrl": "https://code.aipor.cc",
      "isActive": true,
      "createdAt": "2026-05-15T10:00:00Z",
      "note": "额度 100 刀，月底到期"
    },
    {
      "id": "uuid-2",
      "name": "Codex 备用 Key",
      "tool": "codex",
      "apiKey": "sk-xxxxxxxx",
      "baseUrl": "https://gaccode.com/codex/v1",
      "model": "gpt-5.5",
      "isActive": false,
      "createdAt": "2026-05-15T10:00:00Z",
      "note": ""
    }
  ]
}
```

**注意事项**：
- 明文存储意味着 Key 对本机其他用户可见，需在 UI 中向用户说明
- 建议文件权限设置为仅当前用户可读（`chmod 600` on macOS，Windows 通过 ACL 限制）
- 用户可直接编辑或备份 `data.json`，这是明文存储的优势

### 4.3 进程感知与重启引导

**问题**：切换 Key 后，已运行的 Claude Code / Codex 进程不会自动感知新配置。

**解决方案**：

1. 切换完成后弹出提示，引导用户重启目标工具
2. 提供"切换并重启"功能：检测目标进程 PID，发送终止信号后自动重启
3. 进程检测方式：
   - macOS：`pgrep -f "claude"` / `pgrep -f "codex"`
   - Windows：`tasklist | findstr claude`

### 4.4 配置文件原子写入

**问题**：写入 JSON 配置时若中途崩溃，会导致文件损坏，工具无法启动。

**解决方案**：

```
1. 读取原始配置文件
2. 在内存中修改目标字段
3. 写入临时文件（config.json.tmp）
4. 原子重命名替换原文件
5. 修改前自动备份（config.json.bak）
```

### 4.5 跨平台路径处理

需统一处理 `~` 路径展开：

- macOS/Linux：`os.homedir()` + `/`
- Windows：`process.env.USERPROFILE` 或 `os.homedir()` + `\`

建议使用 Node.js 的 `path.join(os.homedir(), '.claude', 'config.json')` 统一处理。

### 4.6 工具安装管理

**需求**：工具需要能够安装 Codex App、Codex CLI、Claude Code、Gemini CLI 等 AI 开发工具。

**各工具安装方式**：

| 工具 | 系统 | 安装命令 / 方式 |
|---|---|---|
| Claude Code | Windows | `irm https://claude.ai/install.ps1 \| iex`（PowerShell） |
| Claude Code | macOS | `curl -fsSL https://claude.ai/install.sh \| bash` |
| Codex App | Windows | 下载 MSIX：`https://get.microsoft.com/installer/download/9PLM9XGG6VKS` |
| Codex CLI | 全平台 | `npm i -g @openai/codex` |
| Gemini CLI | 全平台 | `npm install -g @google/gemini-cli`（需 Node.js 20+） |

**前置检测逻辑**：

```
安装前检测：
1. 检测 Node.js 是否已安装（node --version），版本需 ≥ 20.0.0（Gemini CLI 要求）
2. 检测 npm 是否可用（npm --version）
3. 检测目标工具是否已安装（which claude / where claude）
4. 检测已安装版本，提示是否需要更新
```

**常见安装问题及处理方案**：

| 问题 | 原因 | 解决方案 |
|---|---|---|
| `permission denied` | npm 全局目录权限不足 | 引导用户执行 `npm config set prefix %APPDATA%\npm`，或以管理员身份运行 |
| PowerShell 执行策略错误 | 默认策略禁止运行脚本 | 提示执行 `Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser` |
| 环境变量设置后不生效 | 当前进程未刷新环境 | 提示重启 PowerShell / CMD，或注销重新登录；验证命令：`echo $env:ANTHROPIC_AUTH_TOKEN` |
| `winget` 不可用 | Windows 版本过低 | 降级方案：直接下载 MSIX 安装包并调用系统安装 |

**难点**：
- 安装过程需要实时展示进度和日志输出（子进程 stdout 流式读取）
- macOS 上 `curl | bash` 安装脚本可能需要 sudo，需在 UI 中引导用户授权

## 五、功能设计建议

### 5.1 核心功能（MVP）

- [ ] Key 列表管理：添加、编辑、删除、备注
- [ ] 一键切换：选中 Key 后自动写入对应工具的配置文件 / 环境变量
- [ ] 多工具支持：Claude Code、Codex CLI / App、Gemini CLI（可扩展）
- [ ] 系统托盘：常驻后台，右键快速切换
- [ ] 本地 JSON 存储：数据保存至 `data.json`，明文可读可备份
- [ ] 工具安装：检测并安装 Codex App、Codex CLI、Claude Code、Gemini CLI

### 5.2 进阶功能

- [ ] Key 有效性验证：切换前调用 API 验证 Key 是否有效
- [ ] 使用量监控：接入各平台 API 查询余额/用量
- [ ] 自动切换：余额低于阈值时自动切换到备用 Key
- [ ] 导入/导出：直接导出 `data.json` 备份，或从备份文件导入
- [ ] 多环境支持：区分开发/测试/生产环境的 Key 组
- [ ] 工具更新检测：检测已安装工具的版本并提示更新

## 六、可行性评估

**结论：完全可行，技术风险低。**

| 维度 | 评估 |
|---|---|
| 技术成熟度 | 高，所有依赖库均稳定成熟 |
| 核心操作复杂度 | 低（读写 TOML/JSON 文件 + Key 列表管理） |
| 主要风险点 | Key 安全存储的跨平台实现；工具安装时的权限与环境检测 |
| MVP 开发周期 | Electron 方案约 2~3 周；Tauri 方案约 3~4 周 |
| 扩展性 | 架构上易于扩展支持 Cursor、Windsurf 等更多工具 |
